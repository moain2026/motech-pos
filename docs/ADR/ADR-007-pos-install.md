# ADR-007 — التنصيب الأولي وربط قواعد البيانات (POS_INSTALL) خارج النطاق

- **الحالة:** مقبول (📄 N/A — مغلقة معمارياً)
- **التاريخ:** 2026-07-06
- **الشاشة:** POS_INSTALL (خدمي/Utility، 596 KB)

## السياق

POS_INSTALL شاشة تنصيب لمرة واحدة تُهيّئ بيئة Onyx POS على جهاز/فرع جديد: تُنشئ ~83 جدولاً، تُعرّف db-links للمركز، وتُهيّئ رسائل النظام والنماذج والمساعدة.

- **proof:** `docs/screens/POS_INSTALL.md §1` → "أكثر الجداول استخداماً: `S_MSGS`, `IAS_FORM_NAME`, `FORM_DETAIL`, `IAS_POS_HELP_KEYS`, `IAS_SYS`, `ALL_USERS`" — جداول ميتاداتا/تهيئة، لا بيانات تشغيلية.
- هذه وظيفة **DevOps/تهيئة** لا وظيفة مستخدم نهائي، ومرتبطة بمعمارية Oracle Forms + db-links التي استُبدلت.

## القرار

**لا تُبنى** كشاشة. التنصيب في المعمارية الحديثة إجراء آلي لا شاشة تفاعلية.

## البديل

- **قاعدة البيانات:** هجرات مُرقّمة قابلة لإعادة التشغيل في `db/migrations/` (V001…V027) تُنشئ سكيمة MOTECH_POS كاملة — بديل «إنشاء الجداول». لا db-links: القراءة من ERP عبر مستخدم least-privilege (`MOTECH_RO`) والكتابة في MOTECH_POS.
- **البذر (seed):** بيانات أولية (مستخدمو auth، مصفوفة الصلاحيات V013، إعدادات) تُدار عبر migrations + ملف `auth-users.json`.
- **تهيئة فرع/جهاز جديد:** إجراء تشغيلي موثّق (نشر الحاوية + تطبيق الهجرات + تسجيل الجهاز عبر `admin/machines`) — لا يحتاج شاشة.
- **التوثيق:** `backend/README.md` يوثّق تشغيل الهجرات؛ يُوصى بإضافة قسم «تهيئة فرع جديد» موجز.

## العواقب

- (+) تنصيب مُتكرِّر آلي (idempotent) بدل شاشة يدوية هشّة.
- (+) لا db-links (سطح هجوم/تعقيد أقل).
- (−) يلزم توثيق تشغيلي بدل «شاشة واحدة» — مقبول ومغطّى بالـmigrations.
