# ADR-008 — تحديث/ترقية قاعدة البيانات (POSI014) خارج النطاق

- **الحالة:** مقبول (📄 N/A — مغلقة معمارياً)
- **التاريخ:** 2026-07-06
- **الشاشة:** POSI014 «تحديث قاعدة البيانات» (POSI إدخال، 298 KB)

## السياق

من ملف المساعدة العربي (`poshelp.chm`) الملتقط في `docs/screens/POSI014.md §1`: الشاشة تُطبّق **ترقية سكيمة** (تعديلات جداول/أعمدة عند تحديث نسخة Onyx) و**أرشفة فواتير** قديمة. أي: مهمة صيانة قاعدة بيانات يدوية يشغّلها المسؤول عند ترقية النظام.

- **proof:** `docs/screens/POSI014.md §1` (نص المساعدة: "تحديث قاعدة البيانات … مدخلات النظام").

## القرار

**لا تُبنى** كشاشة. ترقية السكيمة في المعمارية الحديثة تُدار بهجرات إصدارية، لا بشاشة مستخدم.

## البديل

- **ترقية السكيمة:** نظام الهجرات المُرقّمة `db/migrations/V*.sql` (Flyway-style) هو مصدر الحقيقة الوحيد لبنية القاعدة — أي تغيير = هجرة جديدة تُطبَّق بأمر واحد، قابلة للمراجعة في git. هذا يستبدل «تحديث قاعدة البيانات» بالكامل.
- **الأرشفة:** الجزء الوحيد ذو القيمة التشغيلية المتبقّية هو **أرشفة الفواتير القديمة**. لا يلزم شاشة: يُنفَّذ كـ job تشغيلي دوري (partition/جدول أرشيف) عند الحاجة، ويُوثَّق كإجراء ops. بيئتنا الحالية لا تحتاجه (الحجم صغير).

## العواقب

- (+) ترقية سكيمة مُتتبَّعة في git بدل تعديلات يدوية غير موثّقة.
- (+) لا خطر «تشغيل التحديث مرتين» — الهجرات idempotent/مُرقّمة.
- (−) الأرشفة تبقى بنداً تشغيلياً مؤجّلاً (بلا حاجة حالية) — مقبول.
