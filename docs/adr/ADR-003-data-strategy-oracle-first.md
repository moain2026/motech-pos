# ADR-003 — إستراتيجية البيانات: Oracle أولاً + إعادة كتابة المنطق

- **الحالة:** مقبول (Accepted) · 2026-06-29
- **المرجع:** `SCHEMA_OVERVIEW.md` · `PACKAGES_ANALYSIS.md` · `SALES_FLOW.md` · `STANDARDS/04_DATABASE.md`, `13_POS_SPECIFIC.md`

## السياق
النظام الأصلي يحمل بيانات حيّة في Oracle `YSPOS23`:
- **20,569** رأس فاتورة (`IAS_POS_BILL_MST`) + **41,945** سطر (`IAS_POS_BILL_DTL`) + جرد/أصناف فعلية.
- منطق الأعمال **كله** في PL/SQL (58 package): الترقيم (`GET_BILL_NO_PRC`)، الضريبة (`YS_TAX_PKG`)، الخصم (`CLC_DISC_VAT_AMT_PRC`)، الولاء (`POS_POINT_PKG`)، الترحيل (`POS_MOV_TRNS_PKG`).
- النظام **موزّع** (POS فرعي → Main عبر DB links)، والكثير من جداول الدفع/الورديات فارغة محلياً لأنها تُرحَّل للمركز.

سؤالان منفصلان يجب الفصل بينهما بوضوح:
1. **أين تعيش البيانات؟** (Oracle مقابل Postgres)
2. **أين يعيش المنطق؟** (نُعيد استخدام packages PL/SQL مقابل نُعيد كتابته في طبقة التطبيق)

## القرار

### (أ) البيانات: Oracle أولاً، خلف Repository Ports
- في MVP نقرأ/نكتب **نفس جداول `YSPOS23`** عبر `node-oracledb`، بأسماء الجداول/الأعمدة الحقيقية (مثل `IAS_POS_BILL_MST.BILL_NO`, `IAS_POS_BILL_DTL.I_CODE/I_QTY/I_PRICE/VAT_AMT`).
- كل وصول للبيانات خلف **port** في الـ domain (`BillRepository`, `ItemRepository`, `ShiftRepository`...) مع **OracleAdapter**. هذا يجعل الانتقال لـ Postgres لاحقاً مسألة adapter جديد فقط.
- نحترم البنية الموزّعة: قراءة الكميات من `MV_ITEM_AVL_QTY`، واحترام أعلام الحالة (`POSTED`, `HUNG`, `WEB_SRVC_TRNSFR_DATA_FLG`).

### (ب) المنطق: إعادة الكتابة في طبقة التطبيق (لا استدعاء packages كـ black-box)
- **نُعيد كتابة منطق الأعمال في domain/application (TypeScript)** مستندين إلى المنطق المُستخرَج والموثّق في `PACKAGES_ANALYSIS.md` و`SALES_FLOW.md` (وهو منطق مفهوم ومُثبت بالمصدر، لا مُخمَّن).
- **السبب:** المنطق هو القيمة الأساسية للنظام الجديد؛ إبقاؤه في PL/SQL يقيّدنا بـ Oracle ويعيد إنتاج صعوبة الصيانة. كتابته في domain نقي = قابل للاختبار، مستقل عن المحرّك، يطابق Clean Architecture.
- **استثناء انتقالي مُبرَّر:** خلال Strangler-Fig (ADR-005)، يجوز **مؤقتاً** استدعاء بعض إجراءات PL/SQL الناضجة الحرجة (مثل الترحيل `MOV_BILLS_PRC` أو تكامل الفوترة الإلكترونية `GNR_TECH_SOLUTION_PKG.SUBMITDOCUMENT`) كـ adapter، ريثما يُعاد بناؤها، لتقليل المخاطرة. يُوثَّق كل استدعاء كهذا كـ "anti-corruption adapter" مؤقت.

### (ج) جدول القرار: نُبقي في Oracle مقابل نُهاجر
| الفئة | في MVP (Oracle) | لاحقاً (Postgres) |
|------|------------------|--------------------|
| الفواتير (MST/DTL) | قراءة/كتابة Oracle | تُهاجر (immutable facts) |
| الأصناف/الأسعار/الكميات | قراءة من Oracle/MV (السلطة للمركز) | تبقى مرجعية مركزية تُسحب للـ PWA |
| الورديات/الدفع | كتابة Oracle | تُهاجر |
| الضريبة/الفوترة الإلكترونية | adapter لـ PL/SQL مؤقتاً | يُعاد بناؤه كـ adapter قابل للتهيئة لكل دولة |
| الولاء | منطق يُعاد كتابته، الجداول المركزية تبقى مرجعاً | حسب سياسة المركز |

## البدائل المدروسة
| البديل | لماذا رُفض/أُجّل |
|--------|------------------|
| **هجرة فورية كاملة لـ Postgres** | مخاطرة عالية على بيانات حيّة + النظام موزّع متصل بـ Main؛ ننتقل بعد إثبات النظام الجديد. |
| **استدعاء كل packages PL/SQL كـ black-box** | يُبقي القفل على Oracle + يعيد إنتاج تعقيد Onyx؛ يفقد قابلية الاختبار/الاستقلال. |
| **كتابة المنطق دون مرجعية المصدر** | يخالف proof-not-assumption؛ منطق الضريبة/الخصم/الترقيم دقيق وحسّاس مالياً. |

## العواقب
- ✅ لا توقّف، لا فقدان بيانات؛ النظام الجديد يعمل على البيانات الحيّة فوراً.
- ✅ المنطق يصبح ملكنا، مُختبَراً، مستقلاً عن المحرّك.
- ✅ التبديل لـ Postgres = adapter جديد + migration، بلا لمس المنطق.
- ⚠️ خطر اختلاف دقيق بين المنطق المُعاد كتابته والـ PL/SQL الأصلي → نلزم باختبارات **golden** تقارن مخرجات حساب الضريبة/الخصم/الترقيم مقابل عيّنات فواتير حقيقية من Oracle.
- ⚠️ adapters PL/SQL المؤقتة دَيْن تقني مُتتبَّع يجب سداده.
