# ADR-004 — Offline-First / PWA

- **الحالة:** مقبول (Accepted) · 2026-06-29
- **المرجع:** `STANDARDS/13_POS_SPECIFIC.md` (§1,§2) · `02_FRONTEND.md` (§4) · `SALES_FLOW.md` (ملاحظة 1)

## السياق
- النظام الأصلي **offline-first بالتصميم:** `EXTRCT_POS_BILL_PRC` يقبل `SAVE_TYP=0` (offline) حيث **الكاشير يولّد `BILL_NO` محلياً** ثم يُزامَن لاحقاً (مع خطأ صريح `-20904` إن غاب الرقم offline). راجع `SALES_FLOW.md` ملاحظة 1.
- المزامنة على مسارين: ترحيل DB-link + فوترة إلكترونية، مع طابور إعادة محاولة (`POS_SQL_QUEUE`).
- POS يفشل بصمت = خسارة مالية مباشرة. البيع **يجب** أن يكتمل بلا إنترنت.

## القرار
نبني الواجهة كـ **PWA offline-first** بحيث الإنترنت تحسين لا شرط:

- **مصدر الحقيقة المحلي:** IndexedDB عبر **Dexie.js** يحتفظ بـ: الكتالوج (`IAS_ITM_MST`/الأسعار)، الكميات المتاحة (لقطة من `MV_ITEM_AVL_QTY`)، العملاء، الفواتير المعلّقة (يقابل `IAS_POS_HUNG_BILLS`)، **طابور المزامنة append-only**.
- **معرّفات offline:** كل فاتورة/عملية تُولَّد بـ **UUID v7** على الجهاز (مرتّب زمنياً، لا تصادم) كمفتاح عميل، يُربط لاحقاً برقم الفاتورة الرسمي عند المزامنة (يماثل توليد `BILL_NO` المحلي).
- **Idempotency إلزامي:** كل عملية تحمل `clientOperationId` (UUID v7) → الخادم يرفض التكرار عبر `Idempotency-Key` (ADR في API_DESIGN). إعادة المحاولة آمنة بلا ازدواج فاتورة/دفع.
- **Service Worker (vite-plugin-pwa/Workbox):** أصول التطبيق Cache-First؛ الكتالوج/الأسعار StaleWhileRevalidate؛ **عمليات البيع (POST) لا تُخزَّن في SW** — تُكتب في طابور IndexedDB وتُزامن بخدمة خلفية.
- **استراتيجيات التعارض:**
  - الفواتير/المبيعات: **immutable facts** تُضاف فقط — لا تعارض. التصحيح بفاتورة عكسية (مرتجع — يقابل `IAS_POS_RT_BILL`/مردود المبيعات POST002)، لا تعديل.
  - المخزون: **السلطة للخادم** — كل بيع يخصم، الخادم يكشف البيع الزائد وينذر (يماثل `MV_ITEM_AVL_QTY` المركزي).
  - الكتالوج/الأسعار: السلطة للمركز (last-write-wins + رقم إصدار).
- **مؤشر حالة دائم:** online/offline + عدد العمليات غير المُزامنة + آخر مزامنة ناجحة.
- **الترتيب:** ساعة الجهاز غير موثوقة → الترتيب النهائي للخادم؛ طابور بترتيب زمني + retry/exponential backoff + dead-letter.

## البدائل المدروسة
| البديل | لماذا رُفض |
|--------|-----------|
| **Online-only (يتطلب اتصالاً للبيع)** | يعطّل البيع عند انقطاع الشبكة — غير مقبول لـ POS ويخالف تصميم النظام الأصلي. |
| **localStorage للبيانات المهيكلة** | غير مناسب لكميات/طوابير/فهرسة؛ المعيار IndexedDB/Dexie. |
| **ترقيم رسمي offline من الجهاز** | الترقيم القانوني يجب ألا يحتوي فجوات؛ الخادم يختم الرقم الرسمي عند المزامنة، والجهاز يأخذ مرجعاً مؤقتاً. |

## العواقب
- ✅ البيع يعمل كاملاً offline؛ مطابق لسلوك Onyx وأنظف.
- ✅ Idempotency + UUID v7 يلغيان ازدواج الفواتير عند إعادة المزامنة.
- ⚠️ تعقيد المزامنة وحلّ التعارض (يُدار في module `sync` معزول).
- ⚠️ حجم بيانات الكتالوج على الجهاز يحتاج إدارة (تحميل تدريجي/حسب الفرع).
- ⚠️ الفوترة الإلكترونية تتطلب اتصالاً؛ تُؤجَّل في طابور خلفي ولا تعطّل البيع، مع منع الترحيل قبل المزامنة الضريبية (حارس مطابق لـ `MOV_BILLS_PRC -20001`).
