# ADR-005 — الترحيل التدريجي بنمط Strangler-Fig

- **الحالة:** مقبول (Accepted) · 2026-06-29
- **المرجع:** `STANDARDS/01_ARCHITECTURE.md` (§4 Strangler-Fig) · Martin Fowler — StranglerFigApplication

## السياق
استبدال نظام إنتاجي حيّ (Onyx Pro POS، 80 شاشة، بيانات حيّة، موزّع ومتصل بـ Main server) دفعةً واحدة (big-bang rewrite) = مخاطرة عالية على البيع والامتثال الضريبي. النظام الأصلي **لا يُكسَر** (قراءة فقط على الإنتاج — قاعدة المشروع). نحتاج تسليم قيمة تدريجياً مع إثبات كل جزء حيّاً.

## القرار
نتبنّى **Strangler-Fig:** نبني النظام الجديد حول القديم ونحوّل وظيفة تلو الأخرى حتى يموت القديم تدريجياً.

- **نقطة التحويل:** النظام الجديد (Motech POS) يعمل على **نفس بيانات Oracle** (ADR-003) جنباً إلى جنب مع Onyx، فلا انقسام في مصدر الحقيقة أثناء الانتقال.
- **التحويل بالشاشة/الموجة** (راجع `SCREENS_PRIORITY.md`):
  - **الموجة MVP:** الدخول (POSLGN)، فتح وردية (POST027)، **فاتورة البيع (POST001)**، الدفع، إقفال الوردية، التقرير اليومي (POSR001). الكاشير ينتقل للنظام الجديد لدورة البيع الأساسية بينما تبقى الشاشات الإدارية/المعقّدة على Onyx.
  - **الموجة 2:** المرتجعات (POST002/POST006)، المعلّقة (POST003)، العملاء/الولاء (POST020/POST021)، أوامر البيع (POST024)، استعراض الفواتير (POST004/POST017).
  - **الموجة 3:** البيانات الأساسية (POSI*)، الإعدادات (POSS*)، باقي التقارير (POSR*)، المخزون/التحويلات (POST018/019/028/029).
- **Anti-Corruption Layer:** التكامل مع منطق Onyx الناضج الحرج (الترحيل `POS_MOV_TRNS_PKG`، الفوترة الإلكترونية `GNR_TECH_SOLUTION_PKG`) عبر adapters مؤقتة، تُعاد كتابتها لاحقاً (متّسق مع ADR-003).
- **حارس عدم الانحدار:** اختبارات golden تقارن مخرجات النظام الجديد (ضريبة/خصم/ترقيم/إجماليات) مقابل فواتير Onyx حقيقية من Oracle قبل تحويل أي شاشة.

## البدائل المدروسة
| البديل | لماذا رُفض |
|--------|-----------|
| **Big-bang rewrite** | مخاطرة قصوى؛ لا تراجع آمن؛ يوقف البيع عند أي خلل. |
| **Parallel run بقاعدتين منفصلتين** | يخلق مصدري حقيقة + مشكلة مزامنة/توفيق معقّدة. تجنّبناه بإبقاء Oracle مصدراً واحداً مؤقتاً. |
| **تجميد Onyx وتطوير معزول** | يحرم العميل من القيمة التدريجية ويطيل دورة الثقة. |

## العواقب
- ✅ مخاطرة منخفضة؛ تراجع آمن لكل موجة (الشاشة القديمة لا تزال تعمل).
- ✅ قيمة مبكرة (دورة البيع الأساسية أولاً) + بناء ثقة تدريجي.
- ✅ مصدر حقيقة واحد (Oracle) أثناء الانتقال يلغي مشاكل التوفيق.
- ⚠️ تعايش مؤقت بين نظامين (تدريب الكاشير على الانتقال التدريجي + توثيق ما يعمل أين).
- ⚠️ adapters anti-corruption دَيْن تقني مُتتبَّع يُسدَّد عند نضج كل وحدة جديدة.
- ⚠️ يتطلب راية ميزة (feature flag) لكل شاشة محوّلة للتراجع السريع.
